---

## 高风险（立即处理）

### 需求层风险
- [ ] **需求根本矛盾**:技术方案与商业目标不一致
- [ ] **需求不清晰**:关键信息缺失，无法准确评估
- [ ] **需求频繁变更**:未建立变更控制机制
- [ ] **用户需求假设未验证**:基于猜测而非数据
- [ ] **依赖关系未识别**:未发现关键依赖项

### 技术层风险
- [ ] **技术不可行**:方案超出当前技术能力或团队水平
- [ ] **架构设计缺陷**:无法支撑预期规模或质量
- [ ] **关键技术依赖**:依赖不可控的外部技术或服务
- [ ] **技术债务严重**:现有系统质量差，影响新功能
- [ ] **安全漏洞**:存在严重安全隐患

### 资源层风险
- [ ] **关键人员缺失**:核心角色无人担任或能力不足
- [ ] **预算不足**:成本估算明显偏低
- [ ] **时间表不现实**:无法在预期时间内完成
- [ ] **外部资源依赖**:依赖不可控的外部资源

### 商业层风险
- [ ] **商业价值存疑**:ROI为负或无法证明
- [ ] **市场时机错误**:过早或过晚进入市场
- [ ] **竞争劣势**:产品相比竞品无优势
- [ ] **法规合规风险**:违反法律法规或行业规范

---

## 中风险（规划应对）

### 需求层风险
- [ ] **需求理解偏差**:团队对需求的解读不一致
- [ ] **用户场景不明确**:缺乏具体使用场景分析
- [ ] **边界条件未定义**:未考虑边缘情况和异常处理
- [ ] **需求优先级不清晰**:缺乏优先级排序依据

### 技术层风险
- [ ] **技术选型争议**:方案未充分论证
- [ ] **扩展性考虑不足**:未考虑未来增长
- [ ] **性能不达标**:未做性能测试或未达标
- [ ] **技术债务积累**:快速迭代导致质量下降
- [ ] **第三方服务不稳定**:依赖的外部服务可靠性存疑

### 团队层风险
- [ ] **团队协作不畅**:跨部门或团队内部沟通问题
- [ ] **技能不匹配**:部分成员能力不足
- [ ] **知识传承缺失**:关键知识依赖少数人
- [ ] **团队士气低落**:成员积极性不高
- [ ] **决策流程不清晰**:缺乏明确决策机制

### 时间线风险
- [ ] **缓冲不足**:未预留足够应对突发情况
- [ ] **里程碑设置不合理**:关键节点不明确或不可达
- [ ] **依赖任务未识别**:未发现前置依赖
- [ ] **并行工作不充分**:可并行的工作串行化了

---

## 低风险（持续关注）

### 质量层风险
- [ ] **测试覆盖不足**:测试用例不全面
- [ ] **代码质量问题**:未建立代码审查机制
- [ ] **文档缺失**:技术文档和用户文档不完善
- [ ] **监控告警不完善**:缺乏有效的监控手段

### 运营层风险
- [ ] **运营资源不足**:缺乏运营支持
- [ ] **用户培训缺失**:未考虑用户教育
- [ ] **客户支持准备不足**:客服团队未做好准备
- [ ] **数据收集不完善**:缺乏数据埋点和分析

### 战略层风险
- [ ] **长期路线图不清晰**:缺乏长期规划
- [ ] **技术发展方向不明确**:技术选型无长远考虑
- [ ] **组织结构不支持**:组织架构不适应项目需求
- [ ] **文化不匹配**:团队文化与项目要求冲突

## 风险应对策略

### 风险矩阵应用

根据风险发生的**可能性**和**影响程度**，将风险分为四类：

| 影响程度\可能性 | 低 | 中 | 高 |
|----------------|----|----|----|
| **高** | 制定计划 | 立即处理 | 立即处理 |
| **中** | 持续关注 | 制定计划 | 制定计划 |
| **低** | 接受风险 | 持续关注 | 制定计划 |

### 高风险应对策略
1. **避免风险**:调整方案，完全规避风险源
2. **降低风险**:采取措施减少风险发生的可能性或影响
3. **转移风险**:通过外包、保险等方式转移风险
4. **终止项目**:当风险无法接受时，果断停止

### 中风险应对策略
1. **制定详细计划**:明确应对措施和责任人
2. **建立监控机制**:定期评估风险状态
3. **准备应急预案**:风险发生时快速响应
4. **增加缓冲**:在时间和预算上预留余量

### 低风险应对策略
1. **持续监控**:定期回顾风险状态
2. **建立告警机制**:当风险升级时及时通知
3. **记录风险日志**:跟踪风险演变
4. **接受风险**:当应对成本超过风险成本时，选择接受

## 使用指南

### 风险识别流程
1. **组织风险识别会议**:邀请相关方参与
2. **逐项检查清单**:覆盖所有风险领域
3. **补充项目特有风险**:根据项目特点添加
4. **评估风险等级**:使用风险矩阵评估
5. **制定应对计划**:明确措施、责任人、时间

### 风险跟踪管理
1. **建立风险台账**:记录所有识别的风险
2. **定期评审**:每周/每月更新风险状态
3. **告警机制**:高风险升级时立即通知
4. **关闭风险**:风险消除或应对完成后关闭

### 最佳实践
- ✅ 风险识别要全面，不要遗漏关键领域
- ✅ 风险评估要客观，基于数据和经验
- ✅ 应对措施要具体，明确可执行的步骤
- ✅ 责任分配到人，避免责任不清
- ✅ 持续跟踪，风险状态是动态变化的

## 常见陷阱

- ❌ 只关注技术风险，忽视商业和团队风险
- ❌ 风险识别走过场，没有深入分析
- ❌ 风险评估主观，缺乏数据和依据
- ❌ 应对措施空洞，无法真正落地
- ❌ 风险跟踪不到位，风险发生时手忙脚乱
- ❌ 低估"低概率高影响"的黑天鹅事件
- ❌ 忽视风险之间的关联性和传导效应
